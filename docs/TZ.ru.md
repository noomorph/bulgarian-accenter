# bulgarian-accenter — ТЗ на Chrome-расширение (v1)

> **Исторический документ.** Это исходное техзадание на v1. Оно описывает словарь как плоский
> `data/stress-dict.json` на ~94 700 ключей — такого файла больше нет: формат стал front-coded
> (`data/stress-dict.txt`, 422 238 форм), и появился второй проход, выводящий ударения по
> парадигме. Сохранено ради контекста и мотивации решений — **не как описание того, что есть
> сейчас.** Актуальное устройство: [ARCHITECTURE.md](ARCHITECTURE.md).

## Контекст и цель

Пользователь учит болгарский и хочет произносительную подсказку прямо на
веб-страницах: расширение находит болгарский текст (по HTML-атрибуту
`lang="bg"` / `lang="bg-*"`) и проставляет в словах ударения — офлайн, по
готовому словарю, без обращения к серверу и без NLP/резолвинга омографов.

Словарь уже сгенерирован, лежит в `data/stress-dict.json` — расширению
нужно только его использовать. Формат и происхождение словаря описаны ниже.

## Формат словаря

`data/stress-dict.json` — плоский объект:

```json
{ "духове": "ду`хове`", "вятър": "вя`тър", "ветрове": "ветрове`" }
```

- Ключ — слово в нижнем регистре, без ударений.
- Значение — то же слово, где символ `` ` `` (backtick, U+0060) вставлен
  сразу ПОСЛЕ каждой ударной гласной буквы.
- Однослоговых слов в словаре нет намеренно — единственный слог не требует
  пометки, слово просто не будет найдено, и это нормально (не баг).
- ~94 700 ключей, ~1.1% из них содержат **больше одного** backtick, где это
  не составное слово с двумя фонетическими ударениями (напр. `а\`виоа\`с` —
  реально так и произносится), а сплющенная неоднозначность: у источника
  было замечено больше одного места ударения для одной и той же орфографии.
  Причина может быть любой — подлинная вариативность произношения,
  омографы с разным смыслом и разным (в каждом отдельном случае —
  единственным) ударением, либо просто расхождение между 4 объединёнными
  словарными источниками. **Это сознательный компромисс v1**: расширение не
  пытается угадывать/резолвить, какой вариант верный — оно помечает все
  наблюдённые позиции сразу, чтобы пользователь сам свёл это к правилу или
  заглянул в словарь. Не "чини" эту неоднозначность эвристиками — если
  захочется борьбы с ней, это отдельная задача (см. "Возможное развитие").
- Словарь пересоздаётся командой `npm run dict:all` (скачивает публичный дамп
  rechnik.chitanka.info, распаковывает в `.cache/`, применяет исправления и
  запускает `scripts/extract-stress-dict.js`).
  Трогать логику генерации в рамках этой задачи не нужно.

## Функциональные требования

1. **Обнаружение болгарского текста.** Найти все элементы, которые попадают
   под действие `lang="bg"` или `lang="bg-XX"` — с учётом того, что
   `lang` наследуется по дереву DOM: если у элемента нет своего `lang`, но у
   ближайшего предка `lang` болгарский — текст внутри тоже болгарский,
   **кроме** поддеревьев, где встретился элемент с явным `lang` для другого
   языка (там нужно остановиться и не заходить внутрь).
2. **Токенизация.** Внутри найденных текстовых узлов выделять слова —
   последовательности кириллических букв (`[а-яА-ЯЁёЪъ]+` или шире, полный
   диапазон кириллицы). Небуквенные символы (пунктуация, цифры, латиница)
   не трогать.
3. **Подстановка ударения.** Для каждого токена — привести к нижнему
   регистру, поискать в словаре. Если нашли:
   - Преобразовать backtick-нотацию в реальный видимый диакритический знак.
     Рекомендуемый способ — вставить combining acute accent U+0301 сразу
     после ударной гласной (даёт композитный глиф вида "а́" при рендере).
     Альтернатива (если U+0301 плохо ведёт себя с шрифтами/поиском по
     странице) — обернуть ударную гласную в `<span>` с подчёркиванием;
     выбор и обоснование — на усмотрение реализации, но не должно ломать
     выделение текста мышью и copy-paste.
   - Сохранить исходную капитализацию слова (если слово в тексте начиналось
     с заглавной буквы — результат тоже должен).
   - Если слова нет в словаре — оставить как есть, никогда не гадать.
4. **Не трогать:** `<script>`, `<style>`, `contenteditable`, `<input>`,
   `<textarea>`, узлы внутри уже обработанного контента (не обрабатывать
   дважды при повторных проходах).
5. **Динамический контент.** Использовать `MutationObserver` (с debounce)
   для страниц, подгружающих контент после начальной загрузки (SPA, лента
   новостей и т.п.).
6. **Вкл/выкл.** Простой тоггл через иконку расширения — включить/выключить
   обработку на текущей вкладке. Состояние можно не сохранять между
   сессиями в v1 (можно, но не обязательно).

## Нефункциональные требования

- **Manifest V3.**
- **Полностью офлайн** — словарь встроен в пакет расширения, никаких сетевых
  запросов с содержимым страницы наружу. Это принципиально: расширение
  читает произвольные веб-страницы пользователя, ничего из них никуда не
  должно уходить.
- **Производительность.** Словарь (~3.9 МБ, ~95k ключей) грузить один раз
  при инициализации content script (`fetch(chrome.runtime.getURL(...))`),
  держать в памяти как обычный JS-объект (hash lookup, доп. структуры типа
  trie не нужны на этом объёме). Обработка не должна давать заметных
  лагов/jank на обычных страницах.
- **Permissions.** `content_scripts` c `matches: ["<all_urls>"]` (болгарский
  `lang`-атрибут может встретиться где угодно — от Wikipedia до форумов).
  Фоновый service worker в v1, скорее всего, не нужен — вся логика в
  content script.

## Явно вне рамок v1 (не делать, если не попросят отдельно)

- Морфологический анализ, определение части речи, резолвинг омографов по
  контексту (в репозитории `ivanchefy` есть похожая инфраструктура для
  другой задачи — сюда её тащить не нужно, это другой масштаб проекта).
- Отправка текста страницы на сервер / облачный резолвер.
- UI настроек / попап помимо простого вкл-выкл.
- Работа на страницах без `lang`-разметки, даже если текст явно болгарский.
- "Умное" разрешение записей с двойным ударением — оставить как есть (см.
  выше).

## Структура проекта (предлагается, не жёстко)

```
bulgarian-accenter/
  manifest.json
  data/stress-dict.json        # уже сгенерирован
  scripts/extract-stress-dict.js  # генератор словаря, не трогать в рамках этой задачи
  src/content.js               # точка входа content script
  src/accent.js                # normalize/lookup/apply-diacritic, юнит-тестируемый модуль
  icons/
```

## Возможное развитие (не в этой итерации)

- Резолвинг ~1,054 неоднозначных записей: часть из них можно развести на
  "подлинная вариативность" vs "расхождение источников" вручную или через
  внешнюю сверку, если это когда-нибудь станет важным.
- Импорт поля `pronounciation` из дампа (в текущей выборке пусто у всех
  проверенных строк, но не проверено по всем 4 источникам — возможно,
  где-то заполнено и полезно).
- Учитывать регистр целых предложений/аббревиатур отдельно, если найдутся
  проблемные кейсы при реальном использовании.
